# خارطة التنفيذ المبنية على دراسة المنافسين

هذه الخارطة تكمل `006-product-roadmap.md`. لا تعيد المهام التي اكتملت في PR رقم 1،
وتتعامل مع كل مرحلة كـPR مستقل يراجعه علي قبل الدمج.

## قواعد الحسم

- كل تغيير على فرع `codex/` وفي PR مستقل؛ لا دمج تلقائي ولا دفع إلى `main`.
- لا يبدأ تنفيذ المرحلة التالية قبل تحقق شروط قبول السابقة.
- لا تعديل لمحتوى Markdown. أي عملية ملفات تظل صريحة وقابلة للاستعادة.
- لا تبعيات تشغيل npm، ولا bundler، ولا شبكة ضمن المسار الافتراضي.
- الميزة التي لا تساعد شخصًا يقرأ ملفًا موجودًا تحتاج دليل استخدام قبل إدخالها.

## ترتيب العمل

### P0.1 — عقد الأداء والقياس

الهدف: منع الميزات التالية من إبطاء الأساس دون ملاحظة.

- [x] مولد fixture حتمي لملف عربي/ثنائي الاتجاه بحجم 10MB و2,000 ملف صغير.
- [x] قياس زمن أول عرض، وبناء فهرس المجلد، واستعلام البحث، وتحديث ملف واحد.
- [x] تسجيل long tasks وذروة الذاكرة في اختبار Playwright منفصل غير متذبذب.
- [x] تحميل المزايا الثقيلة عند الحاجة، مع degradation واضح إن تجاوز المستند الحد.

القبول:

- يظهر أول ملف قبل انتهاء فهرسة المجلد.
- نتيجة البحث بعد الفهرسة خلال 200ms على fixture المجلد.
- تحديث ملف واحد لا يعيد فهرسة جميع الملفات.
- لا تجميد للمسار الرئيسي يتجاوز 200ms عند تحديث ملف 10MB، أو تظهر حالة معالجة واضحة
  ويظل التحكم في الواجهة مستجيبًا.

الحالة: **DONE** في فرع `codex/performance-contract`. يعالج Web Worker الملفات الأكبر من
2MB، ويعرضها تدريجيًا بأجزاء 96KB. القياس المرجعي المحلي: بحث 11ms، وأطول long task
172ms قبل تصغير الدفعة النهائية إلى ثلاثة أجزاء؛ يفرض الاختبار الآن حد 200ms.

### P0.2 — تقرير سلامة المستند

الهدف: تحويل الروابط المكسورة من مفاجأة عند النقر إلى تقرير قابل للإصلاح.

- [x] تحليل الروابط النسبية وwikilinks والعناوين الداخلية والصور والأصول المحلية.
- [x] حالات: سليم، مفقود، عنوان مفقود، خارج الجذر، غير قابل للفحص، ورابط شبكة غير مفحوص.
- [x] لوحة مختصرة بعدد المشكلات، مع الانتقال إلى موضع كل مشكلة ونسخ المسار.
- [x] تحديث تفاضلي عند تغير الملف المستهدف، من دون إجراء طلبات شبكة.
- [x] استعمال النواة الحالية لحل الروابط حتى لا يختلف التقرير عن سلوك النقر.

القبول:

- fixture يغطي المسافات وUnicode و`../` وaliases والعناوين المكررة والصور.
- لا يقرأ المحلل أي مسار خارج جذر القراءة بعد `realpath`.
- لا ينفذ أي HTTP/DNS للتحقق من رابط خارجي.
- يتطابق تصنيف الرابط مع نتيجة فتحه الفعلية في كل fixture.

الحالة: **DONE** في فرع `codex/document-health`. لا يجري التقرير أي طلب DNS أو HTTP،
ويتحقق من المسار الحقيقي للأصل قبل تصنيفه داخل جذر القراءة.

### P0.3 — نسخ غني يحافظ على الاتجاه

الهدف: نقل ما يقرأه المستخدم إلى Word/Google Docs/البريد دون فقد البنية.

- [x] كتابة `text/plain` و`text/html` في عملية نسخ واحدة عند توفر Clipboard API.
- [x] إزالة أزرار الواجهة والسمات الداخلية والسكريبتات من HTML المنسوخ.
- [x] الحفاظ على العناوين والقوائم والجداول والاقتباس والكود والاتجاه `dir`.
- [x] fallback نصي آمن إذا رفض المتصفح الحافظة الغنية.

القبول:

- اختبار آلي يتحقق من MIME الاثنين ومن خلو HTML من `script` وevent handlers.
- fixture عربي يبدأ بكلمة إنجليزية ويحافظ على اتجاه الفقرة بعد اللصق.
- الجدول والقائمة والكود يحتفظ كل منها ببنيته، ولا تظهر أزرار النسخ داخل الناتج.

الحالة: **DONE** في فرع `codex/rich-copy`. يتحقق Playwright من MIME الفعلي للحافظة،
ومن RTL والبنية الدلالية وخلو HTML من السكريبتات والأحداث وأزرار الواجهة.

### P0.4 — وضع كتاب `SUMMARY.md`

الهدف: قراءة التوثيق والكتب متعددة الملفات كجلسة واحدة دون build.

- [x] اكتشاف `SUMMARY.md` وعرض ترتيب الروابط الهرمي كما هو.
- [x] السابق/التالي وتقدم لكل فصل ونسبة إجمالية.
- [x] تنقل الروابط النسبية داخل متن مع حفظ الرجوع والتقدم.
- [x] إظهار حدود الفصول واسم الملف؛ لا دمج DOM ضخم افتراضيًا.
- [x] تجاهل الروابط الخارجية أو الخارجة من الجذر مع بيان السبب.

القبول:

- fixture مطابق لبنية mdBook/GitBook مع فصول متداخلة وعناوين داخلية.
- غياب `SUMMARY.md` لا يغير سلوك شجرة الملفات الحالي.
- لا يكتب متن أي ملف ولا يشغل mdBook أو أي processor.
- استئناف الكتاب يعيد الفصل والموضع والنسبة الإجمالية الصحيحة.

الحالة: **DONE** في فرع `codex/book-mode`. يفسر متن `SUMMARY.md` للقراءة فقط، ويعرض
كل فصل منفصلًا مع السابق/التالي والتقدم الإجمالي، ولا يشغّل أداة بناء أو يكتب ملفًا.

### P1.1 — قراءة `stdin`

الهدف: `command | matn -` لعرض مخرجات الأدوات وLLM فورًا.

- [ ] قبول `-` مسارًا وحيدًا وقراءة stdin بحد حجم معلن.
- [ ] نسخ المحتوى إلى ملف جلسة خاص بمتن، لا إلى مجلد المستخدم.
- [ ] اسم عرض اختياري وخيار حفظ صريح لاحقًا؛ لا حفظ تلقائي في المصدر.
- [ ] تنظيف ملفات الجلسة القديمة بأمان.

القبول:

- `printf '# عنوان' | matn - --no-open` يفتح جلسة قابلة للقراءة.
- الإدخال الفارغ والضخم والانقطاع ينتج رسائل واضحة ولا يعلق العملية.
- لا يسمح اسم العرض بعبور المسارات، ولا يُعامل المحتوى كـHTML موثوق.

### P1.2 — عرض المصدر للقراءة والمقارنة

الهدف: مساعدة قارئ التوثيق على فهم البنية أو مقارنة التصيير دون إدخال محرر.

- [ ] تبديل rendered/source مع أرقام أسطر وروابط من الكتلة المصيّرة إلى سطر تقريبي.
- [ ] source للقراءة فقط، مع نسخ وتحديد وبحث، بلا textarea أو حفظ.
- [ ] خيار split على الشاشات الواسعة بعد اختبار أداء مستقل.

القبول:

- لا توجد أي API كتابة ملف مرتبطة بالوضع.
- الانتقال من عنوان مصيّر يصل إلى نطاق السطر الصحيح في fixtures.
- RTL في النص المصيّر لا يغيّر LTR الضروري لبنية Markdown والكود في المصدر.

## تجربة مقيدة قبل MCP

لا يُنفذ MCP قبل اكتمال P0 وقياس استعمال Skill و`stdin`. عند إعادة التقييم:

- أداتان فقط: `search_markdown` و`read_markdown`.
- قراءة فقط، `127.0.0.1`، مع token عشوائي، مغلق افتراضيًا، وجذر جلسة واحد.
- حدود للحجم والنتائج، وسجل محلي مختصر، ولا CORS عام.
- threat model واختبارات traversal وCSRF وDNS rebinding قبل الإطلاق.

## قائمة الانتظار، لا خطة حالية

- تصفية قائمة العناوين؛ تحسين صغير بعد قياس صعوبة التنقل.
- autoscroll/WPM وTTS؛ يحتاجان طلبًا واضحًا من قراء النصوص الطويلة.
- مقارنة Git؛ مفيدة للمطورين لكنها لا تخدم كل قارئ، وتحتاج تصميمًا لا يربك الواجهة.
- citations/BibTeX؛ مفيدة أكاديميًا، لكن أضيق من سلامة المستند والكتاب.
- CSS مخصص؛ يعاد بحثه بعد تثبيت أمان التصدير والنسخ الغني.
