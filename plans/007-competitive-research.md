# دراسة المنافسين والفرص التنافسية لمتن

تاريخ الدراسة: 2026-07-11
خط الأساس: `477715a` على فرع `codex/reader-workspace-ux`

## القرار التنفيذي

لا يحتاج «متن» إلى منافسة Obsidian أو Notion في التحرير وقواعد البيانات والمزامنة. الفرصة
الأوضح هي أن يصبح **القارئ المحلي الافتراضي لملفات Markdown، وخصوصًا النصوص العربية
والثنائية الاتجاه ومخرجات الأدوات وLLM والتوثيق متعدد الملفات**.

المزايا الأعلى عائدًا بعد التنفيذ الحالي هي:

1. تقرير سلامة المستند: الروابط والصور والملفات المشار إليها والمراجع الداخلية المكسورة.
2. وضع «كتاب/مشروع» يقرأ `SUMMARY.md` دون بناء أو تعديل الملفات.
3. نسخ غني يضع `text/html` و`text/plain` معًا ويحافظ على RTL والجداول والكود.
4. مسار أداء معلن للملفات الكبيرة والمجلدات الكبيرة، مع قياسات وحدود قبول ثابتة.
5. إدخال CLI من `stdin`، ثم API قراءة محلية صغيرة أو MCP للبحث والجلب فقط بعد تثبيت الأمان.

لا نوصي ببناء محرر، أو رسم معرفي، أو قواعد بيانات، أو مزامنة، أو نظام إضافات، أو دردشة AI.
هذه المنتجات تحل مشكلات مختلفة، وإضافتها ستضعف السبب الذي يجعل «متن» واضحًا وسريعًا.

## نطاق الدراسة ومنهجها

الدراسة لا تدّعي حصر كل تطبيق Markdown في العالم. اختيرت مجموعة تمثيلية تغطي أربع فئات:

- قارئات ومعاينات مباشرة: Marked، Glow، mdBook.
- محررات Markdown ذات تجربة قراءة قوية: Typora، MarkText، Zettlr، VNote.
- مساحات معرفة محلية: Obsidian، Logseq، SiYuan، SilverBullet، QOwnNotes، Joplin.
- إشارة ناشئة واسعة الطموح: Noteriv.

فُحصت الصفحات الرسمية، ثم نُسخت المستودعات المفتوحة نسخًا سطحيًا إلى مجلد مؤقت خارج
المشروع، وفُحصت الوحدات المرتبطة بالعرض والبحث والتنقل والحذف والتكامل. لا تدخل أي شيفرة
من المنافسين في مستودع «متن». بيانات الشعبية تقريبية لحظة الفحص وليست معيار قرار مستقلًا.

## ماذا كشفت الشيفرة، لا صفحات التسويق

### 1. الحذف الآمن منتج صغير مستقل

تنفيذ QOwnNotes لا يعامل «حذف الملف» كاستدعاء `unlink`: لديه أوضاع بلا سلة/سلة نظام/سلة
محلية، وسجل استعادة، وتنظيف تلقائي بعد مدة. هذا يؤيد قرار «متن» الحالي: سلة النظام فقط،
وتحذير الروابط وGit، وعدم السقوط إلى حذف دائم. لا نحتاج قاعدة بيانات سلة محلية الآن؛ نظام
التشغيل يحقق الاستعادة بأقل حالة يملكها التطبيق.

### 2. فصل renderer استثمار طويل الأجل

يفصل Joplin حزمة العرض عن تطبيق سطح المكتب، وتحمّل قواعده frontmatter وMermaid وKaTeX
والتعقيم والروابط. القيمة هنا معمارية: تبقى نواة «متن» النقية قابلة للاختبار، وتُمنع
معالجات DOM من أن تصبح المصدر الوحيد للحقيقة. التنفيذ الحالي بدأ هذا عبر `render-core.cjs`؛
ينبغي استمرار النقل عند إضافة سلامة الروابط والكتاب، لا استيراد renderer المنافس.

### 3. الكتاب ليس «دمج ملفات» فقط

يعامل Marked وmdBook فهرس الكتاب عقدًا للترتيب والتنقل، ويتتبعان الملفات التابعة ويظهران
حدودها. القيمة لمتن هي جلسة قراءة واحدة فوق ملفات كثيرة: فصل واضح للفصول، التالي/السابق،
تقدم إجمالي، وروابط داخلية. لا نحتاج compile أو قالب نشر أو plugins. [2][16]

### 4. التكامل مع الوكلاء يجب أن يكون محدودًا وقابلًا للإيقاف

خادم MCP في QOwnNotes يعرض أداتين فقط: البحث والجلب، ويطلب Bearer token. هذا أفضل من
Runtime API عامة مثل SilverBullet أو عشرات أدوات كتابة الملفات. إذا أضيف MCP لمتن، يكون
قراءة فقط، مرتبطًا بـ`127.0.0.1`، مغلقًا افتراضيًا، ومحصورًا في الجذر المفتوح.

### 5. أداء المستند الكبير ميزة، لا تحسين لاحق

تستخدم SiYuan فهارس FTS5 مستقلة، ويؤخر Marked بعض الخرائط للمستندات الطويلة، ويفصل Glow
الاكتشاف عن العرض. لا يحتاج متن SQLite الآن، لكنه يحتاج fixture ثابتًا لملف 10MB ومجلد
2,000 ملف وميزانية زمنية وذاكرة. من دونها يمكن لكل ميزة جديدة أن تبدو سليمة وتفسد المنتج.

### ما نرفضه الآن

- التحرير داخل متن؛ يضاعف مساحة الأخطاء ويزيل وضوح المنتج.
- backlinks كتجربة مستقلة أو graph؛ الروابط الواردة الحالية كافية للقارئ.
- قواعد بيانات/Dataview/blocks/canvas/flashcards؛ هذه PKM لا قراءة.
- مزامنة وحسابات وتعاون؛ تناقض المحلية وتضيف تشغيلًا دائمًا.
- نظام إضافات أو Runtime API عام؛ عقد توافق وأمن لا تبرره حاجة مثبتة.
- AI chat أو تلخيص عام؛ يمكن للوكلاء قراءة الملفات دون أن يصبح متن واجهة دردشة.
- جلب URL تلقائي؛ يوسع سطح الشبكة وSSRF. يمكن إعادة بحثه بعملية استيراد صريحة ومعزولة.

## استشارة Claude Code باستخدام Fable 5

شُغّل `Claude Code 2.1.206` بالاسم الرسمي الذي قبله CLI: `--model fable`، وبمستوى
`--effort xhigh` ومن دون أدوات كتابة. خلصت الاستشارة إلى نقدين نتبناهما:

1. «قارئ Markdown» وحدها فئة ضعيفة إن لم يكن الفتح الافتراضي أسرع من البدائل.
2. التمايز القابل للدفاع هو العربية/RTL وقراءة مخرجات الأدوات والوثائق، لا عدد المزايا.

واقترحت مراقبة الملفات، والبحث العربي، وoutline، والنسخ الغني، والكتاب. الثلاثة الأولى
موجودة أصلًا في PR 1؛ لذلك لم نكررها في أعلى الخارطة، ورفعنا سلامة المستند وعقد الأداء.
كما أوصت برفض MCP كليًا. نختلف جزئيًا: لا نبنيه الآن، لكن MCP قراءة فقط قد يحل طلبًا
حقيقيًا لعرض ملفات Claude/Codex داخل متن، بشرط قياس الاستخدام وإغلاقه افتراضيًا.

## مخاطر قانونية وهندسية

- يجوز تعلم الأفكار والأنماط العامة. لا تنسخ شيفرة GPL/AGPL إلى مشروع MIT.
- حتى المصادر MIT/MPL لا تُنسخ تلقائيًا؛ الأولوية لإعادة تنفيذ صغيرة متوافقة مع معمارية متن.
- يحتفظ كل PR بسجل المصدر المفاهيمي، ولا ينسب الكود الجديد إلى منافس ما لم يُشتق منه فعلًا.
- Noteriv إشارة سوق حديثة، لا دليلًا على نضج الميزة؛ لا تعتمد ميزة منه دون اختبار مصدرها.
